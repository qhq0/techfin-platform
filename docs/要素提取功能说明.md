# 要素提取功能说明

> **对应需求**：业务需求说明书 — 企业材料上传与信息采集（步骤2：材料解析与提取）
>
> **前置条件**：已完成材料上传与提交（详见 [`上传材料功能说明.md`](上传材料功能说明.md)）
>
> **参考 API**：`api.json`（Postman Collection — dib_api 资料详情接口）

---

## 一、核心业务逻辑

### 1.1 背景

前端提交资料（`POST /techfin/sxd/submit-materials`）成功后，后台系统会自动对上传的文档进行解析和要素提取。由于文档解析是异步过程，前端需要轮询查询提取进度，以决定何时展示提取结果或允许下一步操作。

### 1.2 查询接口

```
GET /techfin/sxd/extract-status/{task_id}
```

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `task_id`（路径变量） | String | 是 | 提交资料返回的任务 ID，格式 `TASK-<32位hex>` |

**成功响应：**

```json
// 全部文档已结束提取（含成功或失败）
{
  "code": 0,
  "msg": "成功",
  "data": {
    "completed": true,
    "pendingDocNames": []
  }
}

// 仍有文档正在提取或等待提取
{
  "code": 0,
  "msg": "成功",
  "data": {
    "completed": false,
    "pendingDocNames": [
      "贵州茅台2025年半年度报告.pdf",
      "商业计划书_v2.pdf"
    ]
  }
}
```

### 1.3 判断逻辑

| 外部 `extractState` | 含义 | 是否计入 `pendingDocNames` |
|---------------------|------|--------------------------|
| `U` | 待执行（Unstarted） | ✅ 是 |
| `I` | 执行中（In Progress） | ✅ 是 |
| `F` | 执行失败（Failed） | ❌ 否（视为已结束） |
| `Y` | 执行完成（Yes） | ❌ 否（视为已结束） |

- **全部完成**：遍历 `application_doc` 表中该 `taskId` 关联的所有文档，逐个调用外部资料详情接口查询 `extractState`。若所有文档的状态均为 `F` 或 `Y`，则判定为完成，返回 `completed: true`。
- **待处理**：只要存在任一文档的状态为 `U` 或 `I`，则返回 `completed: false`，并将这些文档的名称（`docName`）放入 `pendingDocNames` 列表供前端展示。
- **边界情况**：如果 `taskId` 在 `application_doc` 表中无关联记录，抛出 `DOC_NOT_FOUND` 异常。

### 1.4 提取数据查询

当前端轮询到提取状态 `completed=true` 后，调用此接口获取商业计划书各部分的提取文本。

```
GET /techfin/sxd/extract-data/{task_id}
```

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `task_id`（路径变量） | String | 是 | 提交资料返回的任务 ID |

**成功响应：**

```json
{
  "code": 0,
  "msg": "成功",
  "data": {
    "extractData": [
      { "tableName": "dib_manage_company_profile", "text": "深圳市WG药业有限公司（简称“WG药业”）成立于2015年……" },
      { "tableName": "dib_director_keyresume", "text": "王美女士，深圳大学商务管理专业……\n李鹏博士，17年高端药物制剂研发经历……" },
      { "tableName": "dib_manage_business_and_products", "text": "深圳市WG药业有限公司定位为专注儿童处方药……" },
      { "tableName": "dib_manage_business_circumstance", "text": "深圳市WG药业有限公司定位为专注儿童处方药的企业……" },
      { "tableName": "dib_company_qualification", "text": "公司荣获国家高新技术企业……" },
      { "tableName": "dib_manage_progressiveness_description", "text": "WG药业构建了“自研创新+合作引进”双轮驱动……" },
      { "tableName": "dib_manage_competitive_advantages", "text": "首先，公司构建了顶尖的全球化管理团队……" },
      { "tableName": "dib_manage_development_strategy", "text": "深圳市WG药业有限公司确立了……" },
      { "tableName": "dib_manage_y_industry_analysis", "text": "中国儿童用药市场呈现出增速快、潜力大……" }
    ]
  }
}
```

#### 判断逻辑

1. 根据 `taskId` 从 `application_doc` 表中查询 `businessType = 1455619597903695872`（商业计划书）的文档列表。
2. 对每个文档，遍历预设的 9 个 `tableName`，逐个调用外部提取数据查询 API：
   - `dib_manage_company_profile` — 公司介绍
   - `dib_director_keyresume` — 实际控制人及团队简介
   - `dib_manage_business_and_products` — 主营业务和产品
   - `dib_manage_business_circumstance` — 经营情况介绍
   - `dib_company_qualification` — 资质认证与荣誉
   - `dib_manage_progressiveness_description` — 研发成果与转化情况
   - `dib_manage_competitive_advantages` — 竞争优势介绍
   - `dib_manage_development_strategy` — 发展战略情况
   - `dib_manage_y_industry_analysis` — 行业发展情况
3. 每个 `tableName` 对应外部 API 响应中不同的文本字段（见下方字段映射表），后端提取后按 `tableName` 聚合。
4. 对于 `dib_director_keyresume`，该表可能返回多个人员记录，将各记录的 `resume` 字段以换行符连接合并为一个 `text`。
5. 按固定顺序返回所有 9 个 `tableName` 的 `text`，即使某表提取结果为空也保留条目（`text` 为空字符串）。

#### 字段映射规则

| tableName | 外部响应 data 中的字段 | 说明 |
|-----------|----------------------|------|
| `dib_manage_company_profile` | `companyProfileText` | 公司介绍全文 |
| `dib_director_keyresume` | `resume` | 多个人员简历，按换行合并 |
| `dib_manage_business_and_products` | `businessAndProductsText` | 主营业务和产品介绍 |
| `dib_manage_business_circumstance` | `text` | 经营情况描述 |
| `dib_company_qualification` | `text` | 资质认证描述 |
| `dib_manage_progressiveness_description` | `progressivenessText` | 研发成果描述 |
| `dib_manage_competitive_advantages` | `competitivenessText` | 竞争优势描述 |
| `dib_manage_development_strategy` | `strategyText` | 发展战略描述 |
| `dib_manage_y_industry_analysis` | `text` | 行业发展分析 |

> 外部 API 的 `data` 返回 JSON 数组，数组中每条记录包含上述字段之一。后端根据 `tableName` 取对应字段的值，多条记录时按换行合并。

## 二、处理流程

```
① 前端轮询 → GET /techfin/sxd/extract-status/{task_id}

② 后端处理 → 根据 taskId 从 application_doc 表查询所有 doc_id
         → 遍历每个 doc_id，调用外部资料详情 API
         → 聚合所有文档的 extractState

③ 结果判定 → 全部为 F 或 Y → completed: true
         → 存在 U 或 I   → completed: false + pendingDocNames

④ 提取数据查询（completed=true 后调用）：
         → 后端根据 taskId 查 business_type=1455619597903695872 的文档
         → 对每个文档遍历 9 个 tableName，调用外部提取数据查询 API
         → 按字段映射提取文本，聚合后返回 {tableName, text} 列表

⑤ 前端展示：
     completed=true  → 调用 GET /sxd/extract-data/{task_id} 获取提取结果并展示
     completed=false → 显示加载动画 + 列出待处理文档名称
```

## 三、外部 API 调用

### 资料详情 `GET /api/extract/open/doc/get/{doc_id}`

```
Headers:
  c1-token: <从 application.properties 配置读取>

Response 200:
{
  "success": true,
  "code": "1",
  "message": "操作成功",
  "data": {
    "id": "1522204279569448960",
    "docName": "3.贵州茅台2025年半年度报告（公开信息）.pdf",
    "projectId": "1521841328915996672",
    "attId": "1522204270320672768",
    "docTypeId": "1273757631042949120",
    "extractState": "Y",
    "classifyState": "20",
    "convertState": "Y",
    "verifyState": "N",
    "auditState": "Y",
    "machineCompletionRate": 100.0,
    "completionRate": 100.0
  }
}
```

> 本功能只关注 `extractState` 字段，其余状态字段（`classifyState`、`convertState` 等）由外部系统内部流转。

### 调用方式

- **HTTP 方法**：`GET`
- **鉴权**：请求头 `c1-token`，值统一在 `api.default-token` 配置
- **URL 拼接**：`api.doc-detail-url/{doc_id}`（配置中的 `api.doc-detail-url` 不含末尾的 `/doc_id`）
- **响应解析**：统一使用 `ExternalResponse` 包装，通过 `getDataAs(DocDetailData.class)` 转换 data 字段

---

### 提取数据查询 `POST /api/extract/open/doc/queryData`

```
Headers:
  c1-token: <从 application.properties 配置读取>
  Content-Type: application/json

Request Body:
{
  "docId": 1522659418314964992,
  "tableName": "dib_manage_company_profile"
}
```

**Response 200：**
```json
{
  "data": [
    {
      "company_profile_text": "深圳市WG药业有限公司（简称“WG药业”）成立于2015年……",
      "doc_id": "1522659418314964992",
      "doc_name": "1.深圳市WG药业有限公司商业计划书（脱敏）.pdf",
      ...
    }
  ],
  "success": true,
  "code": "1",
  "message": "操作成功"
}
```

> 响应 `data` 为 JSON 数组，每项包含一个文本字段（字段名因 `tableName` 而异，详见 [字段映射规则](#字段映射规则)）。

**请求参数**

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `docId` | Long | 是 | 文档 ID，来自 `application_doc.doc_id`，类型为数值 |
| `tableName` | String | 是 | 表名，对应商业计划书提取的不同部分 |

**调用方式**

- **HTTP 方法**：`POST`，请求体为 JSON
- **鉴权**：请求头 `c1-token`，值统一在 `api.default-token` 配置
- **URL**：`api.doc-query-data-url`（完整地址，不含路径变量）
- **请求体序列化**：DTO 使用 `@JsonNaming(LowerCamelCaseStrategy.class)` 确保 `docId` 字段不因全局 SNAKE_CASE 配置被序列化为 `doc_id`
- **响应解析**：使用 `ExternalResponse` 包装，通过 `getDataAsList(ExtractQueryDataRecord.class)` 转换 data 数组

## 四、数据模型

### application_doc 表（文档明细表）

查询提取状态时，通过 `task_id` 索引获取该任务下的所有 `doc_id`。

| 字段 | 类型 | 说明 |
|------|------|------|
| `doc_id` | VARCHAR(64) (PK) | 资料批量新增返回的文档 ID |
| `task_id` | VARCHAR(64) | 关联 `application_record.task_id`（已建索引 `idx_task_id`） |
| `business_type` | VARCHAR(32) | 业务类型（docTypeId） |
| `report_date` | VARCHAR(10) | 财报报告日期 |

### 关键 DTO

| 类 | 位置 | 说明 |
|----|------|------|
| `ExtractStatusResponse` | `model/.../dto/response/` | 提取状态查询响应体：`completed` + `pendingDocNames` |
| `ExtractDataResponse` | `model/.../dto/response/` | 提取数据查询响应体：`extractData`（`{tableName, text}[]`） |
| `DocDetailData` | `model/.../dto/external/` | 外部资料详情 API 的 data 映射：`id`, `docName`, `extractState` |
| `ExtractQueryDataRecord` | `model/.../dto/external/` | 外部提取数据查询 API 的 data 数组元素，汇总所有可能文本字段 |
| `ExtractQueryDataRequest` | `model/.../dto/external/` | 外部提取数据查询 API 的请求体：`docId` + `tableName` |

## 五、配置项

```properties
# 外部资料详情接口地址（不含 doc_id，调用时拼接）
api.doc-detail-url=${api.base-url}/api/extract/open/doc/get
# 外部提取数据查询接口地址（POST，完整路径）
api.doc-query-data-url=${api.base-url}/api/extract/open/doc/queryData
```

| 属性 | 说明 |
|------|------|
| `api.doc-detail-url` | 资料详情接口基地址，调用时在末尾拼接 `/{doc_id}` |
| `api.doc-query-data-url` | 提取数据查询接口完整地址（POST JSON） |

## 六、错误码

| 错误码 | 触发条件 | HTTP 状态码 |
|--------|---------|------------|
| `DOC_NOT_FOUND` | `taskId` 在 `application_doc` 表中无关联记录 | 400 |
| `DOC_DETAIL_FAILED` | 外部资料详情接口调用失败（网络异常 / 非成功响应） | 400 |
| `DOC_QUERY_FAILED` | 外部提取数据查询接口调用失败（网络异常 / 非成功响应 / docId 格式无效） | 400 |

## 七、与上传材料功能的关系

```
上传材料功能                         要素提取功能
─────────────────                  ────────────────
POST /upload-attachment     →      （前置步骤）
POST /submit-materials      →      生成 taskId
                                  GET /extract-status/{task_id}
                                  （轮询直到 completed=true）
                                  GET /extract-data/{task_id}
                                  （completed=true 后获取提取结果）
```

> 提取状态查询和提取数据查询是上传材料的后续流程，均共享 `application_doc` 表作为数据桥梁。提取数据仅在 `completed=true` 后可用，此后前端可进入下一阶段（如实控人确认等）。
